# Review Booster テンプレート方針

**重要**: この方針は今後の開発で必ず守ること。

---

## テンプレートの定義

- **テンプレート = BARVEL用の Review Booster**
- 新しいレビューブースター（新規店舗）を作成するときは、**このテンプレートをコピーしたうえで**変更を加える。
- **アンケート項目は固定**とする。テンプレートと同じ項目構成をベースにし、店舗ごとに「表示する／しない」や文言だけを変える。

---

## テンプレートの内容（BARVEL KOZA ベース）

- **UI**: `src/app/[storeId]/page.tsx` の共通レイアウト（評価 → ご本人は？ → 来店タイプ → 良かったポイント → 誰と → 推しスタッフ → 生成 → コピー・Googleマップ）
- **アンケート項目（固定）**:
  1. 評価（星1〜5）
  2. ご本人は？（男性 / 女性）
  3. 来店タイプ（地元 / 観光 など）
  4. 良かったポイント（複数選択）
  5. 誰と（友達 / 同僚 / 恋人 / 一人）
  6. 推しスタッフ（いれば・任意入力）
- **設定**: `src/config/stores.ts` の **`_template`** がマスターテンプレ（コピー元）。同じ構造で新規店舗を追加し、`features.*.enabled` や `keywords.options` などだけを店舗用に変更する。
- **コピー用ファイル**: [docs/stores.template.example.md](stores.template.example.md) にコピペ用の設定ブロックあり。`/_template` で同じUI・全質問項目を確認できる（一覧には出さない）。

---

## 新規店舗を追加するときの手順

**フロー全体・決めること一覧・Gemini 用指示文**は [新規店舗の作り方.md](新規店舗の作り方.md) にまとめてある。以下は要約。

1. **設計を決める**
   - [新規店舗の作り方.md](新規店舗の作り方.md) の「決めておく必要があること一覧」を確認し、Gemini にユーザーと協議させて最終仕様を出力させる（同 doc の「Gemini に渡す指示文」を利用）。
2. **テンプレートをコピーする**
   - `stores.ts` の **`_template`** ブロック、または **docs/stores.template.example.md** のコードブロックをコピーし、新規店舗用のキー（例: `"my-store"`）で `stores` に追加する（UIは共通のため、新規ページをゼロから作らない）。
3. **変更を加える**
   - 最終仕様に従い、店舗ID・名前・ロゴ・GoogleマップURL・テーマ色
   - キーワードの文言（アンケートの「良かったポイント」の選択肢）
   - 必要に応じて `companion` / `gender` / `visitType` / `staffName` の `enabled: false` で非表示
   - `promptContext` で口コミ生成用の店舗説明を設定
   - `actions.ts` にキーワード言い換えや店舗分岐を追加（同 doc の「実装時の参照」を参照）。
4. **アンケート項目は固定のまま**
   - 項目の追加・削除はテンプレートと大きく変えず、**ON/OFF と文言の差し替え**で対応する。

## 厳守するルール

以下の2項目は、AI・開発者ともに**絶対に守る**こと。レイアウト崩れやプロンプトの分岐地獄を防ぐためである。

### 1. UIレイアウト原則（単一ページ構成）

- **ルール（セクション全体）:** 単一ページ構成では、各セクションを縦に積み重ねる（例: `<div className="space-y-6">`）。項目の並び順は固定とする（下記）。
- **例外（良かったポイント）:** 「良かったポイント（キーワード・複数選択）」は **デフォルトで 2 列（`grid-cols-2`、4 択なら 2×2）** とする。一覧性と縦スクロール量のバランスのため。ステップ式レイアウト（本家）と同じグリッド方針に揃える。
- **5 個以上のキーワード:** 店舗で選択肢が 5 個以上になる場合は、折り返し・読みやすさを優先し **`grid-cols-1`（1 列）** に変更してよい（`[storeId]/page.tsx` の当該 `ToggleGroup` のみ）。
- **その他の質問:** 性別・来店タイプ・誰と、など **2 択系のトグルは引き続き 2 列**でよい（既存どおり）。
- **禁止・注意:** 3 列以上のグリッドや、同一画面内で用途の異なる複雑な多段レイアウトは避ける（保守性・タップ領域のため）。
- **並び順の固定:** 「評価 → ご本人は？（性別） → どんな来店でしたか？ → 良かったポイント → 誰と来ましたか？ → 推しスタッフ」の固定順とする。

### 2. AIプロンプト（actions.ts）の「コアロジック共通化」原則

- **ルール:** 新規店舗を追加し `actions.ts` に設定を記述する際、口コミの生成ロジック（トーン＆マナー、感情表現、ハシゴ酒などの文頭バリエーション、絵文字の量）は、**必ずベースとなるテンプレート（BARVEL等）と完全に同一の構造・指示を用いること**。
- **禁止事項:** 店舗ごとに「独自の感情語のルール」や「独自の文体バリエーション」を勝手に追加・変更してはならない（保守性の低下を防ぐため）。
- **変更が許可される変数:** 新規店舗の設定で変更してよいのは、以下の3点のみとする。
  1. `storeName`（店舗名）
  2. `locationContext`（名護、コザなどの地名・立地文脈）
  3. `keywordVariations`（良かったポイントの選択肢に対応するキーワードと言い換え）

---

## 覚えておくこと

- 全く同じコピーを用意した上で、**そこに変更を加えていく**形にする。
- 新しいレビューブースターを作る = テンプレ（BARVEL）をベースにした設定の追加と、必要なら専用レイアウトの分岐（CEBUOCTO のような 1 ページ構成など）のみ。
